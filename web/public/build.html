Use cases

Four shapes out of the box.

The kernel is always the same: committed rules → deterministic replay → on-chain signature. The shape you pick is just a semantic label.

🎟

Raffle

Select a winner (or N winners) from a weighted participant list. Seed derived from resolve slot.

🎁

Loot

Roll a weighted outcome table — legendary / rare / common. Same kernel, different label.

🪂

Airdrop

Weighted eligible-list selection. Experimental multi-winner output with per-address caps.

🔮

Prediction Market

Pre-commit possible outcomes and weights before the event resolves. No oracle required.

How it works

Three steps. Three commits. Zero hidden state.

Each step produces a public artifact. Each artifact references the previous one. You never need to trust the engine — the chain of hashes is the proof.

1

Define rules

Write a JSON config: participants, weights, outcome ids, selection mode. The SDK hashes it and gives you an artifact_hash.

$ vre init raffle.json
2

Commit on-chain

Push the hash to Solana. The VRE program stores it against a runtime_id. Once committed, the ruleset cannot be edited.

$ vre resolve --config raffle.json --program-id <ID>
3

Share and verify

You get a signature. Paste it at verifiableoutcome.online/verify or run vre verify. Anyone does this — not just you.

$ vre verify --sig <SIG> → MATCH / OK

Programmatic API
// The same engine, from JS import { VRE } from "verifiable-outcome-sdk"; const vre = new VRE({ cluster: "devnet", programId: "3b7T…XkBq", }); const runtime = await vre.commit(config); const outcome = await vre.resolve(runtime.id); // Independent check — anyone can run this const result = await vre.verify(outcome.sig); // { status: "MATCH", code: "OK", … }

Invariants

What VRE actually guarantees.

1

Pre-commitment

The ruleset hash is on-chain before the resolve transaction. Slot ordering is enforced by the validator, not by VRE.

2

Determinism

The selection function is a pure mapping from (seed, rules) to outcome. Same inputs, same output — forever.

3

Public seed

Seed is derived from a future slot's block hash — unknowable at commit time, public after resolution.

4

No oracles

Every input the verifier needs is already on-chain. Public RPC is the only dependency.
